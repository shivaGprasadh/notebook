OWASP Top 10 Security Risks

Published on April 28, 2025

The Open Web Application Security Project (OWASP) Top 10 is a standard awareness document for developers and web application security practitioners. It represents a broad consensus about the most critical security risks to web applications. This article explores the current OWASP Top 10 list, explaining each risk and providing guidance on how to mitigate them.

1. Broken Access Control

Access control enforces policies such that users cannot act outside their intended permissions. Failures typically lead to unauthorized disclosure, modification, or destruction of data.

Common vulnerabilities include:

  • Violation of the principle of least privilege
  • Bypassing access control checks by modifying the URL or HTML page
  • Permitting viewing or editing someone else's account by providing its unique identifier
  • Accessing API without access controls for POST, PUT, and DELETE
  • Elevation of privilege (acting as a user without being logged in or acting as an admin when logged in as a user)

Prevention measures:

  • Implement server-side access controls for all sensitive functions and resources
  • Deny access by default, except for public resources
  • Use rate limiting to minimize automated attack impacts
  • Disable web server directory listing
  • Log access control failures and alert administrators

2. Cryptographic Failures

Failures related to cryptography often lead to exposure of sensitive data. This category includes inadequate encryption of sensitive data and common mistakes in implementing encryption.

Common vulnerabilities include:

  • Transmitting sensitive data in clear text
  • Using deprecated or weak cryptographic algorithms
  • Using default or weak cryptographic keys
  • Not enforcing encryption using directives like HTTP Strict Transport Security (HSTS)
  • Using passwords as cryptographic keys without proper key derivation

Prevention measures:

  • Classify data processed, stored, or transmitted by an application
  • Apply controls according to the classification
  • Don't store sensitive data unnecessarily
  • Use strong up-to-date algorithms, protocols, and keys
  • Encrypt all sensitive data at rest and in transit
  • Use strong adaptive and salted hashing functions for passwords

3. Injection

Injection flaws, such as SQL, NoSQL, OS, and LDAP injection, occur when untrusted data is sent to an interpreter as part of a command or query.

Common vulnerabilities include:

  • User-supplied data is not validated, filtered, or sanitized
  • Dynamic queries or non-parameterized calls without context-aware escaping
  • Hostile data used within object-relational mapping search parameters
  • Hostile data directly used or concatenated in dynamic queries, commands, or stored procedures

Prevention measures:

  • Use a safe API that avoids using interpreters entirely
  • Use parameterized queries or prepared statements
  • Use LIMIT and other SQL controls within queries to prevent mass disclosure of records
  • Use input validation with allowlist approach

4. Insecure Design

Insecure design refers to flaws in the design and architecture of applications, as distinct from implementation flaws. It involves failures in threat modeling, secure design patterns, and proper security requirements.

Common vulnerabilities include:

  • Lack of business risk profiling in the software or system being built
  • Failure to determine the security requirements and controls needed
  • Establishing trust boundaries and access controls based on untrusted inputs
  • Excessive trust in client-side validation

Prevention measures:

  • Establish a secure development lifecycle with AppSec professionals
  • Use threat modeling for critical authentication, access control, business logic, and key flows
  • Integrate security language and controls into user stories
  • Use secure design patterns and reuse hardened components
  • Write unit and integration tests to validate all critical flows are resistant to the threat model

5. Security Misconfiguration

Security misconfiguration is the most commonly seen issue, often resulting from insecure default configurations, incomplete or ad hoc configurations, open cloud storage, misconfigured HTTP headers, and verbose error messages.

Common vulnerabilities include:

  • Missing appropriate security hardening across the application stack
  • Unnecessary features enabled or installed
  • Default accounts and passwords still enabled and unchanged
  • Overly detailed error messages that reveal sensitive information
  • Not using HTTP security headers

Prevention measures:

  • Develop a repeatable hardening process that makes it fast and easy to deploy
  • Use a minimal platform without unnecessary features
  • Review and update configurations appropriate to all security notes, updates, and patches
  • Segment application architecture with effective separation and security controls
  • Automate verification of configurations and settings in all environments

6. Vulnerable and Outdated Components

This vulnerability occurs when components, such as libraries, frameworks, and other software modules, run with the same privileges as the application but contain flaws that attackers can exploit.

Common vulnerabilities include:

  • Not knowing the versions of all components you use
  • Using outdated or vulnerable software, including OS, web/application server, database, applications, APIs, and all components
  • Not scanning for vulnerabilities regularly
  • Not fixing or upgrading the underlying platform, frameworks, and dependencies
  • Not securing the components' configurations

Prevention measures:

  • Remove unused dependencies, unnecessary features, components, files, and documentation
  • Continuously inventory versions of both client-side and server-side components
  • Only obtain components from official sources over secure links
  • Monitor for libraries and components that are unmaintained or don't create security patches
  • Establish a patch management strategy

7. Identification and Authentication Failures

Authentication failures involve functions related to user identity, authentication, and session management that are often implemented incorrectly.

Common vulnerabilities include:

  • Permitting automated attacks like credential stuffing
  • Permitting brute force or other automated attacks
  • Permitting weak or well-known passwords
  • Using weak or ineffective credential recovery processes
  • Using plain text, encrypted, or weakly hashed passwords
  • Missing or ineffective multi-factor authentication

Prevention measures:

  • Implement multi-factor authentication to prevent automated credential stuffing, brute force, and stolen credential reuse
  • Do not ship or deploy with default credentials
  • Implement weak password checks
  • Ensure registration, credential recovery, and API pathways are hardened against account enumeration attacks
  • Limit or increasingly delay failed login attempts

8. Software and Data Integrity Failures

Software and data integrity failures relate to code and infrastructure that does not protect against integrity violations. This includes insecure CI/CD pipelines and using untrusted plugins, libraries, or modules.

Common vulnerabilities include:

  • Using plugins, libraries, or modules from untrusted sources or CDNs
  • Using an insecure CI/CD pipeline without verifying code or creating immutable artifacts
  • Using unsigned or unverified auto-update functionality without sufficient integrity verification
  • Not checking updates and patches for digital signatures

Prevention measures:

  • Use digital signatures or similar mechanisms to verify the software or data is from the expected source
  • Ensure libraries and dependencies are consuming trusted repositories
  • Ensure your CI/CD pipeline has proper segregation, configuration, and access control
  • Verify that unsigned or unverified code is not loaded into your application

9. Security Logging and Monitoring Failures

This category helps detect, escalate, and respond to active breaches. Without logging and monitoring, breaches cannot be detected.

Common vulnerabilities include:

  • Auditable events, such as logins, failed logins, and high-value transactions, are not logged
  • Warnings and errors generate no, inadequate, or unclear log messages
  • Logs of applications and APIs are not monitored for suspicious activity
  • Logs are only stored locally
  • No alerting thresholds or response processes in place

Prevention measures:

  • Ensure all login, access control, and server-side input validation failures can be logged with sufficient context
  • Ensure logs are generated in a format that log management solutions can easily consume
  • Ensure log data is encoded correctly to prevent injections or attacks on logging systems
  • Implement audit trails with integrity controls for high-value transactions
  • Establish effective monitoring and alerting

10. Server-Side Request Forgery (SSRF)

SSRF flaws occur whenever a web application is fetching a remote resource without validating the user-supplied URL. It allows attackers to coerce the application to send a crafted request to an unexpected destination.

Common vulnerabilities include:

  • Accept URLs or file paths as input for server-side requests without validation
  • Not enforcing URL schema, port, and destination with an allowlist
  • Not having a deny-by-default policy that blocks access to all internal-only functionality

Prevention measures:

  • Sanitize and validate all client-supplied input data
  • Enforce the URL schema, port, and destination with an allowlist
  • Use network segmentation to separate sensitive internal services from the DMZ
  • Use an application gateway to filter all outbound traffic
  • Disable HTTP redirections and prevent DNS rebinding attacks with proper DNS configuration

Conclusion

The OWASP Top 10 provides a valuable framework for understanding the most critical web application security risks. By addressing these vulnerabilities, organizations can significantly improve their security posture and protect their applications from common attack vectors.

Remember that security is an ongoing process, not a one-time event. Regular testing, monitoring, and updating of security practices are essential to maintaining a robust security posture against evolving threats.
